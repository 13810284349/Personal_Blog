很多团队第一次做 AI Agent，容易把注意力放在模型本身：选哪个大模型、提示词怎么写、能不能调用更多工具。

这些当然重要，但真正决定 Agent 能不能在企业里长期运转的，往往不是“它有多聪明”，而是三个更朴素的问题：团队成员能不能自然地找到它？它能不能稳定地替不同项目执行任务？它学会的能力能不能被下一支团队复用？

这张“企业 AI Agent 协作平台”的架构图，回答的正是这三个问题。它把平台拆成三层：用户交互层、AI Agent 平台层、Skills 技能体系。每一层都只解决一类问题，合起来才像一个可持续扩展的组织级 AI 基础设施。

## 第一层：把 Agent 放进团队已经在用的地方

最上面一层是用户交互层，入口选择了飞书。

这不是一个装饰性的选择。企业内部工具最怕的不是功能不够，而是使用路径太长。如果一个团队成员每次都要打开新网站、登录新系统、学习新界面，再去描述自己的问题，那么 Agent 还没开始工作，使用意愿已经被消耗了一半。

把入口放在飞书里，意味着团队成员可以在日常沟通流里直接与 Agent 对话。架构图里强调了三点：不需要安装额外工具、打开飞书即可用、实时流式响应。这三个点加在一起，本质上是在降低使用门槛。

对用户来说，他不是在“操作一个 AI 平台”，而是在群聊或私聊里找一个熟悉的同事帮忙。这个感受很关键。Agent 越像一个自然存在的团队角色，它越容易被纳入真实工作流，而不是停留在演示场景里。

## 第二层：Agent 不是一个万能助手，而是一组团队角色

中间一层是 AI Agent 平台层，也是整张图的核心。

飞书传来的自然语言请求，会通过 WebSocket 长连接进入平台，再由 `claude-to-im` 桥接引擎转给 Claude Agent SDK。这个桥接层的意义，是把即时通讯里的消息、用户、上下文和流式返回，转换成 Agent 运行时可以理解和执行的任务。

更重要的是，图里没有把所有请求都塞给一个“超级 Agent”，而是为不同团队和业务场景拆出了多个专属 Agent：

- Flys Scout 面向社交产品，负责 AI 内容分析、运营、Badcase 管理、链路追踪、热度运营等能力。
- FYE Scout 面向游戏项目，覆盖 Unity SVN、代码审查、Sentry、云效流水线等研发流程。
- NS SRE Scout 面向运维和稳定性，处理 K8s 开发环境、生产环境、日志排查等问题。
- Operations Scout 面向运营，连接小红书、微信公众号、Android 等平台相关能力。

这种设计的价值在于，它承认企业里的问题从来不是同一种问题。

研发团队问的是：“这次发布为什么失败？”SRE 问的是：“这个 Pod 为什么重启？”运营团队问的是：“这条内容的数据为什么异常？”如果这些问题都交给同一个 Agent，它就会越来越像一个堆满插件的工具箱，什么都有一点，什么都不够贴合。

专属 Agent 的好处，是可以拥有自己的身份、上下文、工作目录和项目知识。架构图里写得很清楚：每个 Agent 独立进程、独立飞书 Bot、独立工作目录。这意味着它们可以共享平台能力，但不会互相污染上下文；可以继承公共技能，但又保留团队自己的业务判断。

企业里的 Agent，不应该只是“会聊天的软件”。它更像一个被分配到团队里的数字成员：有岗位，有权限，有边界，也有自己负责的那一小块问题。

## 第三层：Skills 是组织经验的沉淀方式

底层是 Skills 技能体系。这里是整个平台最值得细看的部分。

如果说 Agent 是团队里的角色，那么 Skills 就是它能调用的工作方法。图里把 Skills 分成两类：全局 Skills 和项目 Skills。

全局 Skills 是所有 Agent 都能共享的基础能力，例如飞书文档、飞书项目、数据库查询、K8s 管理、GitHub、数据分析、日志查询、向量搜索。这些能力不属于某个单一项目，而是很多团队都会反复用到的公司级能力。

项目 Skills 则更贴近具体业务场景，例如 Sentry 错误监控、云效流水线、代码审查、Badcase 测试集管理、链路追踪、热度运营、小红书和微信接口、自进化能力。它们往往带有很强的团队语境：某个项目的发布流程是什么，错误等级如何判断，数据异常应该先看哪里，Badcase 该如何归档。

这套分层很重要。

如果所有能力都做成全局能力，平台会很快变得臃肿，团队特有知识也容易被过度泛化。如果所有能力都只留在项目内部，又会导致重复建设：每个团队都重新写一遍查日志、查文档、查数据库、跑流水线的方法。

全局 Skills 解决“大家都一样”的部分，项目 Skills 解决“我们不一样”的部分。一个新项目接入时，只需要创建飞书 Bot，再编写自己的项目 Skill，就能自动继承全部全局能力。这让平台扩展不再依赖一次次从零搭建，而是像搭积木一样持续复用。

## 为什么这套架构适合企业内部

这张图最清晰的设计判断，是它没有试图把 AI Agent 做成一个孤立产品，而是把它做成企业协作系统的一部分。

第一，入口在协作工具里。用户不用离开工作现场，Agent 才有机会参与真实问题。

第二，运行时有隔离。每个 Agent 有自己的进程、Bot 和工作目录，减少了上下文串扰，也方便按团队管理权限和责任。

第三，能力以 Skills 形式沉淀。提示词、工具调用、排查步骤、业务规则都不只是一次性对话，而可以变成可维护、可复用、可演进的能力单元。

第四，平台允许按需扩展。图里保留了“你的项目 Agent”的位置，说明这不是一个固定编制，而是一套可以让新团队接入的机制。

这也是企业 AI 基础设施和个人 AI 工具的差别。个人工具追求的是马上帮我完成一次任务；企业平台追求的是让很多人、很多团队、很多项目持续获得同一套基础能力，同时保留各自的业务差异。

## 真正的难点：不是调用工具，而是定义边界

从技术上看，让 Agent 调用工具并不稀奇。更难的是定义它能在什么上下文里调用、能替谁调用、调用之后如何追踪、出错时由谁负责。

例如，日志查询和数据库查询都是高价值能力，但它们天然需要权限边界。K8s 管理和云效流水线可以提升效率，也可能造成真实生产影响。Sentry、Badcase、链路追踪这些能力看起来只是查询工具，但它们背后涉及团队如何判断优先级、如何复盘问题、如何关闭任务。

因此，一个成熟的 Agent 平台不能只问“能不能接上这个工具”，还要问：

- 这个能力应该做成全局 Skill，还是项目 Skill？
- Agent 执行前需要哪些权限确认？
- 哪些操作只能读，哪些操作可以写？
- 任务执行过程是否可追踪、可复盘？
- 业务知识更新后，Skill 如何同步迭代？

这些问题决定了 Agent 是一个可控的生产力系统，还是一个难以治理的自动化入口。

## 一条轻量的接入路径

架构图底部给出了一个很克制的接入流程：创建飞书 Bot，编写项目 Skill，即刻上线。

这条路径看起来简单，但背后的平台能力已经把复杂度提前吸收掉了。团队不需要自己重新实现即时通讯连接、流式响应、Agent 运行时、全局工具、基础权限和工作目录隔离。它只需要描述清楚自己的业务场景，并把特有流程沉淀成 Skill。

这让接入从“开发一个 AI 应用”变成“为团队配置一个 Agent”。前者需要完整项目投入，后者更像是把现有工作方法写进一个可执行的协作伙伴里。

当平台上已经有 11 个运行中的 Agent、12 个全局 Skills、19 个项目 Skills，并且保持 24/7 在线时，它的价值就不再只是单点效率提升，而是形成了一种组织能力：每新增一个 Agent，都会继承已有基础设施；每新增一个 Skill，都可能反过来增强多个 Agent。

## 结语：让 AI 成为团队结构的一部分

好的企业 AI Agent 平台，不是把每个人都推向一个更复杂的工具台，而是把智能能力安静地放进团队已经在使用的协作结构里。

飞书入口解决触达，Agent 平台解决运行，Skills 体系解决复用。三层合在一起，才让 Agent 从一次性的对话助手，变成可以被团队分工、可以被项目扩展、可以被组织沉淀的基础设施。

如果说过去的自动化是在替人执行固定流程，那么这一代 Agent 平台更像是在帮助团队保存和复用判断：谁负责什么，遇到问题先看哪里，哪些动作可以自动做，哪些决策必须交还给人。

这才是企业 AI Agent 最值得期待的地方。它不是让工作消失，而是让那些重复解释、重复查找、重复串联工具的部分慢慢退后，把人的注意力还给真正需要判断的地方。
