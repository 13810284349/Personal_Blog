我建议接下来开发 **P0：API 可观测性与错误追踪基础设施**。

理由：读完三份文档和架构演进路线后，读者侧功能基本已经很完整了：归档、搜索、目录、相关文章、封面、SEO、评论审核、限流、通知直达都已落地。现在剩下的最高优先级风险不是再加一个页面，而是生产环境出问题时能不能快速定位。架构文档也明确说最大风险在密钥管理、评论滥用和生产环境观测；评论滥用已处理，密钥轮换偏外部操作，最适合继续开发的是观测能力。

文章归档页（推荐先做）

路由：/archive
按年份/月份分组展示所有已发布文章
加入导航和 sitemap
不引入新依赖，风险低，和博客气质很贴

站内搜索

等文章数量到 10-20 篇后再做更划算
可用 Pagefind 生成静态搜索索引
现在只有少量文章，搜索的收益还不如归档页
评论防刷 / rate limit

如果线上开始有垃圾评论，再优先做
可以按 IP hash、slug、时间窗口限制提交频率
这是安全维护型功能，不是当前最能提升阅读体验的功能
所以我的判断是：先开发 /archive 文章归档页。它小、稳、可见，而且会自然接上后续的搜索、年度总结、时间线等能力。


我建议下一步做：文章详情页的上一篇 / 下一篇导航。

理由很简单：现在已经有首页、标签、归档，站点的“入口”基本齐了；下一步最自然是改善读完一篇后的流动感。这个功能风险低、不引入依赖，也很贴个人博客气质。

可以这样做：

在 /posts/[slug] 底部展示“上一篇 / 下一篇”
顺序沿用 getPublishedPosts() 的发布时间倒序
只包含已发布文章，过滤草稿
文案保持克制，比如：
上一篇
下一篇
显示文章标题和日期
移动端单列，桌面左右并排
不需要新增 API、数据库、环境变量
备选顺序我会排成这样：

上一篇 / 下一篇导航：最推荐，轻量且立刻提升阅读体验。
站内搜索 /search：文章数量变多后很有用，但现在 3 篇文章略早。
文章详情页目录 TOC：适合长文，尤其技术文章多起来后。
相关文章推荐：可基于标签做，但比上一篇/下一篇更容易显得“产品化”。



我建议下一步做：站内搜索。

理由很朴素：你现在已经有首页、归档、标签、文章详情、上一篇/下一篇，浏览路径基本完整了；下一块最能提升读者体验的是“我记得你写过某个词，但不知道在哪篇”。

可以做得很轻，不引入数据库：

构建期从 getPublishedPosts() 生成搜索索引，只包含已发布文章。
搜索范围：标题、描述、标签、正文摘要。
页面：新增 /search，加入主导航。
交互：前端本地搜索，不新增 API、不碰 Supabase。
结果展示：标题、日期、摘要命中片段、标签。
移动端保持单列，风格延续现在的极简阅读感。
备选功能我会排在后面：

文章目录 TOC：对长文很有用，但当前文章数量和长度还不算最迫切。
相关文章推荐：可以按标签匹配，但刚做完上一篇/下一篇，先做搜索更互补。
评论审核体验增强：偏后台效率，读者侧价值没有搜索直接。



我建议下一步做：文章目录 / 标题锚点。

理由很直接：你现在已经有搜索、归档、标签、相邻文章导航，站点的“发现”能力基本成型了。下一步最该补的是单篇长文的阅读体验。像《把 AI Agent 接进团队日常》这种长文章，读者需要能快速扫结构、跳到章节、复制某一节链接。

建议范围：

文章详情页自动提取 h2/h3 生成目录
标题自动带稳定 id，支持直接跳转
桌面端目录放在右侧 aside，和现有互动区保持克制
移动端目录折叠或放在正文前，不挡阅读
不新增 API、不碰 Supabase，纯静态 HTML 行为
样式延续现在的极简阅读感
第二梯队可以排：

文章目录 / 标题锚点：最值得先做。
搜索页体验增强：如清空按钮、键盘焦点、结果摘要更精准。
OG 图片 / 分享卡片：提升外部分享质感。
评论审核页增强：批量审核、状态筛选、跳转原文。
我的判断：先做目录。它会立刻改善长文阅读，而且和现有架构最顺。





我建议下一步做：相关文章推荐。

理由很顺：现在单篇文章已经有目录、锚点、上一篇/下一篇，阅读闭环基本完整；下一步最能提升浏览深度的是在文章底部根据标签推荐 2-3 篇相关已发布文章。

实现也比较克制：

纯静态 HTML，不新增 API、不碰 Supabase。
复用 getPublishedPosts() 和文章 tags。
只推荐已发布文章，排除当前文章。
按共享标签数量优先，再按发布时间倒序。
放在正文之后、评论区之前，和相邻文章导航保持同一套极简阅读风格。
文章不足或没有相关标签时不渲染。
这个功能比“阅读进度条 / 目录当前章节高亮”更适合作为下一步，因为它不需要新增前端滚动脚本，风险更低，也更符合现在这个博客的静态、安静路线。



我建议下一步做：文章封面与社交分享图支持。

理由很顺：cover 字段已经在内容模型里预留了，但现在还没真正用起来。把它补完能提升文章分享体验和搜索/社交预览质量，而且可以保持纯静态，不新增 API、不碰 Supabase。

可以做成很克制的一版：

cover 存在时，文章详情页在标题区后展示一张正文宽度封面图
BaseLayout 输出 og:image、twitter:image
列表页先不强行加封面，避免破坏极简阅读风格
没有 cover 时完全保持当前样子
文档同步更新 AGENTS.md 和 API 技术文档
优先级我会排成这样：

文章封面 / OG image 支持：已有字段，收益明显，风险低。
文章系列 / 专题合集：适合以后内容多了以后组织长线主题。
搜索页体验增强：比如按标签过滤、结果计数、清空按钮。
评论审核页优化：批量审核、状态筛选体验，但会更偏后台工具。


我会建议下一步做：文章结构化 SEO 支持。

理由很顺：你刚做完封面和社交分享图，下一层就是让文章在搜索引擎和社交平台里“被正确理解”。

具体可以做：

/posts/[slug] 输出 Article JSON-LD
BaseLayout/文章页补充 article:published_time、article:modified_time、article:author、article:tag
RSS 可评估是否带上 cover/enclosure
文档同步写清楚 SEO/meta 规则
这个功能不破坏现有极简阅读风格，也不需要 Supabase/API/客户端脚本，和当前架构很贴合。第二优先级我会放“内容发布体验”：加一个 npm run new-post 脚本生成带 frontmatter 的文章模板。



建议下一步做：Pagefind 全文搜索升级。

理由很顺：你已经完成了 SEO、RSS、sitemap、自动部署这些“被外部发现”的能力，下一步就该提升站内“被读者找到”的能力。现在 /search 是构建期手写轻量索引，只覆盖标题、描述、标签和截断摘要；文章多起来后，全文搜索会更有价值。

Pagefind 很适合这个博客，因为它是纯静态搜索，不需要 Supabase、不需要新 API、不需要服务器；官方也说明它会在构建后生成静态搜索 bundle，并且 npx pagefind 包含对中文/日文索引的扩展支持：
Pagefind docs / Pagefind installation

建议开发范围：

保留 /search 页面和极简风格
用 Pagefind 替换当前手写搜索索引
构建后生成 dist/pagefind/
只索引文章正文区域，排除导航、footer、评论区、点赞区
搜索结果展示标题、摘要、发布日期、标签
无新增数据库、无新增 API、无新增环境变量
更新 README.md、AGENTS.md、技术文档里的搜索规则和验证方式
开工前先把当前 3 个文档改动提交推送一下，然后我会把 Pagefind 作为下一张任务卡来做。



我建议下一步做：评论审核后台增强。

理由很顺：现在“被发现”和“被找到”已经补齐了，读者进来之后最可能产生的新成本就是评论审核。当前 /admin/comments 是轻量可用版，下一步把它做成更顺手的维护工具，会直接提升长期运营体验。

建议开发范围：

审核页支持按状态筛选：pending / approved / rejected
支持关键词搜索：昵称、正文、文章 slug
支持按时间倒序浏览
显示评论所属文章标题和链接
增加一键通过 / 拒绝 / 恢复待审
保持 noindex, nofollow
不新增数据库表，优先复用现有 /api/admin/comments
必要时扩展现有后台 API query 参数
更新技术文档和 AGENTS.md
备选也可以是 自动 OG 分享图，但我会把它排在评论后台之后。OG 图提升外部分发质感，评论后台提升日常维护效率；以当前博客阶段，后者更实用。


我建议下一步做：评论反垃圾与提交限流。

理由：评论审核后台已经顺手了，但真正会抬高长期运营成本的下一件事是垃圾评论。现在公开 POST /api/comments 已开放，已有 honeypot 和基础校验，但还缺少服务端防刷。

建议范围：

同一 IP hash + 同一文章短时间内限制提交频率
同一 IP hash 短时间全站提交数量限制
重复正文检测，避免刷屏
可选敏感词/垃圾词拦截，先用环境变量配置，不新建后台
错误提示保持中文友好
不暴露 IP，不新增浏览器端密钥
优先复用现有 blog_comments、ip_hash、created_at
更新 API 技术文档和 AGENTS.md
这是很自然的下一步：先让评论能管，再让垃圾评论不容易进来。



下一步做：评论待审通知。

理由很直接：现在评论已经有审核后台和反垃圾限流，但新评论进入 pending 后，管理员仍然需要主动想起来打开 /admin/comments。这会变成新的长期运营成本。待审通知能把“有评论需要处理”推到你面前，收益高、改动小，也不破坏现在的极简架构。

建议范围：

新评论成功进入 pending 后，服务端发送一条通知。
用环境变量配置 webhook，例如 COMMENT_NOTIFY_WEBHOOK_URL，不新增后台。
通知内容只包含文章标题/链接、昵称、正文摘要、审核入口，不包含邮箱、IP hash、原始 IP。
通知失败不影响评论提交，只记录服务端错误。
更新 API 文档和 AGENTS.md。
部署后验证：提交一条测试评论，确认后台可见且通知到达。
备选功能我会排在后面：文章专题/系列页、公开阅读统计页、草稿预览。但按当前节奏，评论待审通知是最顺手、也最能继续降低维护成本的一步。


下一步做：审核通知直达评论。

理由：你刚加了“待审通知”，但通知里的审核入口现在只能到 /admin/comments。真正使用时，下一步摩擦会变成：打开后台、输入 token、在列表里找对应评论。把通知入口做成“直达这条评论”，收益很高，改动也很小。

建议范围：

通知里的 reviewUrl 改成类似：
https://macondo-co.netlify.app/admin/comments?status=pending&q=<commentId>
后台审核页初始化时读取 URL 参数：
status=pending|approved|rejected
q=关键词或评论 ID
/api/admin/comments?q=... 支持按 id 精确搜索，保留现有昵称/正文/slug 搜索。
后台加载后能直接看到这条评论，不新增登录系统、不新增数据库表。
更新 API 文档和 AGENTS.md。
这会把刚做好的通知闭环补完整：收到通知 → 点开 → 直接审核。
比继续做大功能更划算，也符合现在这个博客“轻、静态优先、低维护”的方向。


下一步做：审核页的直达体验收尾。

现在通知能直达 /admin/comments?status=pending&q=<commentId> 了，但如果当前浏览器没有存过 token，页面还是停在“输入 token，然后手动点载入评论”。下一步可以把这个摩擦再抹掉：

带 status/q 参数打开后台时，显示提示：输入审核 token 后将载入这条评论。
token 输入后自动触发载入，或按回车即可载入。
当 q 是评论 ID 时，列表文案用“评论 ID”而不是“关键词”，更清楚。
可选：审核通过/拒绝后，如果当前是评论 ID 直达页，提示“已处理这条评论”。
这个改动很小，但会把链路从“收到通知 → 点开 → 输入 token → 点按钮 → 审核”变成“收到通知 → 点开 → 输入 token → 审核”。收益很贴近刚做的功能。

如果想做稍大一点的下一步，我的第二推荐是：评论审核后的通知反馈，比如 webhook 发一条“评论已通过/已拒绝”的内部通知。但那会引入更多通知事件，维护成本略高，不如先把审核入口体验打磨完。
